# RFC: `crib init`

**Status:** Draft
**Goal:** Generate a working devcontainer config from project conventions so users can `crib init && crib up` without writing JSON.

## Problem

Every new project requires manually creating `.devcontainer/devcontainer.json` (and often a Dockerfile) before crib can do anything. This is a barrier for:

- Users new to dev containers who don't know the config format
- Users who want to quickly containerize an existing project
- Teams onboarding new members to a project that doesn't have a devcontainer config yet

The config is mostly boilerplate that can be derived from project conventions (Gemfile = Ruby, go.mod = Go, package.json = Node, etc.).

## Design

### Usage

```bash
cd my-project
crib init        # detect project type, generate config
crib init --type ruby   # skip detection, use ruby template
crib init --dry-run     # print what would be generated
```

### Detection

Walk the project directory looking for known files. First match wins (ordered by specificity):

| File(s) | Type | Base image | Features |
|---------|------|------------|----------|
| `Gemfile` + `config/application.rb` | `rails` | `ruby:<version>` | node (if `package.json`), postgres client |
| `Gemfile` | `ruby` | `ruby:<version>` | |
| `go.mod` | `go` | `golang:<version>` | |
| `package.json` + `next.config.*` | `nextjs` | `node:<version>` | |
| `package.json` + `nuxt.config.*` | `nuxt` | `node:<version>` | |
| `package.json` | `node` | `node:<version>` | |
| `pyproject.toml` or `requirements.txt` | `python` | `python:<version>` | |
| `Cargo.toml` | `rust` | `rust:<version>` | |
| `pom.xml` or `build.gradle*` | `java` | `eclipse-temurin:<version>` | maven or gradle |
| `composer.json` | `php` | `php:<version>` | |
| `mix.exs` | `elixir` | `elixir:<version>` | |
| `*.sln` or `*.csproj` | `dotnet` | `mcr.microsoft.com/dotnet/sdk:<version>` | |

If nothing matches, fall back to a generic `debian` base with no language tooling.

### Version detection

Extract the language version from project files where possible:

- **Ruby**: `.ruby-version`, `Gemfile` (`ruby "3.3.0"`), `.tool-versions`
- **Node**: `.node-version`, `.nvmrc`, `package.json` (`engines.node`), `.tool-versions`
- **Go**: `go.mod` (`go 1.23`)
- **Python**: `.python-version`, `pyproject.toml` (`requires-python`), `.tool-versions`
- **Rust**: `rust-toolchain.toml`, `rust-toolchain`

If no version is detected, use a sensible default (latest LTS or stable).

### Generated files

**Always: `.devcontainer/devcontainer.json`**

Minimal, commented config. Example for a Rails project:

```jsonc
// Generated by crib init. See https://fgrehm.github.io/crib/
{
  "name": "my-project",
  "image": "ruby:3.3",
  "features": {
    "ghcr.io/devcontainers/features/node:1": {}
  },
  "postCreateCommand": "bundle install",
  "remoteUser": "vscode"
}
```

**If needed: `.devcontainer/Dockerfile`**

Only generated when the base image needs customization that can't be expressed as features (e.g., system packages specific to the project). Most projects should get by with just `image` + `features`.

**Always: `.cribrc`**

With detected cache providers:

```
cache = bundler, npm, apt
```

**Never generated:**
- Docker Compose files (too project-specific, can't guess service topology)
- `.dockerignore` (use existing or none)
- Lifecycle hooks beyond obvious ones (e.g., `bundle install` for Ruby, `npm install` for Node)

### Behavior

- **Won't overwrite**: if `.devcontainer/devcontainer.json` already exists, print a message and exit. Use `--force` to regenerate.
- **Won't overwrite `.cribrc`**: same behavior, `--force` to replace.
- **Interactive mode** (future): prompt for choices when detection is ambiguous. For v1, just pick the best guess and let the user edit.
- **Prints what it did**: show the generated files and suggest `crib up` as next step.

### Compose detection

If `docker-compose.yml` or `compose.yaml` already exists in the project root, `crib init` should:

1. Ask (or detect) which service is the dev container
2. Generate a devcontainer.json that references the compose file instead of an image
3. Skip Dockerfile generation (compose owns the build)

This is the one case where interactive prompting (or a `--service` flag) is useful in v1.

## What it does NOT do

- **No runtime detection**: doesn't check if Docker/Podman is installed or running
- **No `crib up`**: just generates files, user runs `crib up` separately
- **No template registry**: templates are compiled into the binary, not fetched from a registry
- **No per-org templates**: teams that want shared configs should use `extends` (when implemented) or a shared git repo

## Implementation

### Package structure

```
internal/
  init/
    detect.go      # project type detection (file existence checks)
    version.go     # language version extraction from project files
    template.go    # devcontainer.json / Dockerfile / .cribrc generation
    templates/     # embedded template files (go:embed)
cmd/
  init.go          # cobra command, flags, output
```

### Templates

Use `text/template` with `go:embed`. Each project type has:

- A `devcontainer.json` template (with conditional sections for features, hooks)
- An optional `Dockerfile` template
- A list of recommended cache providers

Templates produce valid JSON with comments (jsonc). Keep them minimal and
well-commented so users understand what each field does.

### Phases

**Phase 1: Single-language projects**
- Detection for Ruby, Go, Node, Python, Rust
- Version extraction from common files
- devcontainer.json + .cribrc generation
- `--type` override, `--dry-run`, `--force`

**Phase 2: Framework detection**
- Rails, Next.js, Django, etc.
- Smarter postCreateCommand (e.g., `bin/setup` for Rails)
- Feature selection based on framework needs (e.g., Node for Rails asset pipeline)

**Phase 3: Compose integration**
- Detect existing compose files
- `--service` flag or interactive prompt
- Generate compose-referencing devcontainer.json

**Phase 4: Interactive mode**
- Prompt for choices when detection is ambiguous
- "Would you like to add PostgreSQL?" style questions
- Still works non-interactively with flags

## Open questions

1. **mise/asdf `.tool-versions`**: should we detect all tools from this file and add matching features/images? Or just the primary language?

2. **Dockerfile vs image+features**: when should we generate a Dockerfile instead of relying on features? Current thinking: only when the project has system-level dependencies that no feature covers.

3. **`remoteUser`**: default to `vscode` (devcontainer convention) or detect from the base image? Official language images use `root`, devcontainer images use `vscode`.

4. **Existing Dockerfile**: if the project already has a `Dockerfile`, should `crib init` reference it in the devcontainer config? Or ignore it and use an image? The Dockerfile might be for production, not development.

5. **Monorepos**: projects with multiple languages (e.g., Go backend + React frontend). Detect the primary language? Generate a multi-feature config? Punt to compose?
